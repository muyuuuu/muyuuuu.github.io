Git与Github的使用

发表于 2019-11-16 更新于 2020-05-01 阅读次数：
本文字数： 5.8k 阅读时长 ≈ 5 分钟

突如其来的兴趣，想把自己写的一些看着不错的代码放到github上，供大家一起交流与分享。毕竟自己之前只会git clone。

今日更新的原因是：居然发现专业的好多人不会使用Git和Github，尽管他们大一就开始写代码，写的很好，但不会使用Git就很离谱，我还在怀疑他们的项目是如何交流and编写的，遂整理常用的Git和Github方法到此。

要声明的是：本文的目标并非对Git和Github一无所知的小白，纯小白看了本文更加会一头雾水。而是有一定的Git和Github使用经验。适用于：说是小白吧，但有点意思，说是精通吧，但又差点意思。

前提是自己已经安装了Git并测试可以使用，安装方法略。

日前

第一次创建

首先需要在Github上完成的工作，新建一个仓库，不建议勾选初始化README.md文件和创建LICENSE等选项，因为这会和本地的仓库造成冲突。假设我新建的仓库名称为：A Beautiful PyQt5 Interface.

首先: 创建某个文件夹，里面放上想要提交上去的文件. git init 完成初始化.
其次: git add . 添加所有文件，这个时候可以随意的插入readme文件
然后: git commit -m "code by LanLing" 添加代码的描述信息
之后: git remote add origin https://github.com/muyuuuu/A-Beautiful-PyQt5-Interface.git 不要忘记 .git
最后: git push -u origin master 完成提交

以后的更新

cd 到对应的文件夹

git add .
git commit -m 'include some information' 增加修改信息，一定要修改，否则会失效
git push 即可完成修改

最终结果

https://github.com/muyuuuu/A-Beautiful-PyQt5-Interface

结束语

可能这只是暂时的用法，至少目前我还分不清branch还有master的概念，也没有和其他人一起开发大项目的经验，日后可能会作出改变，甚至是一些观念，多走几步看看。就像昨天我还不会用git，但是今天来看，上述的命令我已经记住和熟练使用了。谁知道呢。

更新

今日，仔细研读了一番Git的使用方法(人傻就要多学习)，又有了新的收获，遂整理如下(指令的速查手册)。当然，多动手实践才是硬道理。纸上得来终觉浅，绝知此事要躬行。

Github是Git托管的一种网页端的服务，Git是一种版本管理神器，即使不用Github，也可以考虑使用Git管理自己的代码和文件。管理的好处？太多了：

  • 是不是遇到过文件忽然丢失，想恢复但恢复不了？Git管理起来就没这个烦恼；
  • 毫不容易实现了一个功能，老师却要一个月前的代码，好不容易删除改好了老师又要一个月后的代码。Git就可以自由穿梭在任何时间点，复原任何时间段的文件。

Git 常用指令

基本操作

  • git init 初始化本地仓库，生成.git文件，存储管理当前目录的数据。
  • git status 查看仓库状态，没有可提交文件或未追踪(untracked)文件。
  • git add 可加文件名或git add . 添加全部文件到暂存区，此时不再显示未追踪文件。
  • git commit -m "comment" 提交修改，而后显示没有可提交文件的状态。或git commit，打开编辑器(操作类似vim)修改添加内容，#为注释，所以内容不要写在# 后面。会生成commit的hashvalue。
  • git log 查看提交日志，包括提交时间，提交作者，联系方式和介绍。也可以git log --pretty=short显示简单信息。也可git log filename 也可以查看指定文件的日志。git log -p 查看本次提交与上次提交的区别，git log -p filename 可查看对应文件的前后区别。
  • git diff 查看当前目录和最新提交之间的差别。在git add .后，git diff显示当前目录和暂存区(.git)的差别，但没有差别。所以此时查看与上一次提交的区别时：git diff HEAD，这里的HEAD是指向当前分支的最新一次提交的指针。

分支操作

用于完成多人开发的并行作业。分支应该集中实现单一主题和功能，除此之外不进行任何作业。即使创建的分支在开发的过程中遇到了BUG，也应该创建新的分支修正。

  • git branch显示分支列表，*后面为此时所处的分支。默认为master主干分支，没有开发到一半的代码，可随时让人查看。
  • git branch test 创建test分支。
  • git checkout test 切换到test分支，在test分支中修改的内容不会显示在master分支中。
  • git merge合并分支。假设子分支的功能开发完毕，切换到主分支，git merge --no-ff branchname创建合并提交，而后输入提交的信息即可。
  • git log --graph 以图表的形式查看分支的合并情况。
  • 当有俩个分支时，主分支有test.md文件，子分支也有test.md文件，这两个文件不同。虽然操作不影响，但在合并分支时会发生冲突，需要手动处理冲突文件然后才能合并。

更改提交

  • git reset 回溯历史版本。借助分散仓库(每个人都可以fork，修改自己的仓库不会影响他人的仓库)的优势，可以不影响其他仓库的前提下对历史版本进行操作。根据累计提交的历史记录，可根据目标时间点的哈希值回溯到指定状态，git reset --hard hashvalue
  • git reflog推进历史。即发现回溯过头，可以用git reflog查看操作日志，找出历史时间点的哈希值，git reset --hard hashvalue即可。就像给定一个时间机器的各个时间点，可自由穿梭。这些时间点来源于每次的git commit。
  • git commit --amend，修改上次提交的内容。如第一次commit时commen填写错误，执行本命令即可。
  • git rebase -i 压缩历史，第一次git commit提交功能代码，第二次git commit修改了其中文件的错别字等小错误。将修改的提交和上次提交的两次提交合并为一次提交，此时git rebase -i HEAD~2会选定当前分支最新提交和上次提交，删除第二次commit的哈希值，保存并退出。此时第二次提交从历史中抹去，第一次提交的哈希值也会更改。（然后git push -f origin master，origin代表远程仓库标识名，master表示要推送的分支）

远程仓库使用

Github是Git托管的一种服务，将本地的Git仓库在远程展示。

推送

  1. 首先在Github上创建仓库；
  2. git remote add origin ....设置为本地仓库的远程仓库，并将远程仓库的名称设置为origin；
  3. git push -u origin master，将当前分支的内容推送给远程仓库origin的master分支，-u参数会使在将来git pull时，本地仓库的当前分支可以直接获取origin的master分支内容，不必费力的设置参数。
  4. git push -u origin branch-test，推送branch-test分支。

获取

  1. git clone 获取远程仓库，默认处于master分支，将origin设置为远程仓库的标志符；
  2. git pull 克隆后可以拉取远程仓库master分支的最新内容；
  3. git branch -a查看本地仓库和远程仓库的所有分支；
  4. git checkout -b branch-test origin/branch-test 获取远程仓库的子分支，-b为创建本地分支的名称，origin/branch-test为分支来源。
  5. 用户A切换到branch-test分支完成工作后，git push即可推送子分支。
  6. 用户B可以git pull origin branch-test获取对应分支在远程仓库的最新内容，这样用户A,B就完成了协同开发。如果两人修改同一部分的源代码，为减少冲突的发生，应该频繁的push和pull。

Github 使用

  • 搜索窗口：输入代码片段或用户，就能搜索到与之相关的信息(也可在Github上整体搜寻)。
  • Gist：一些没必要录入仓库的小代码，展示小的Demo。
  • Stars：类似点赞。

仓库

首先打开任意的仓库。

上侧

  • star：仓库的点赞人数；
  • watch：该仓库的相关信息会在All activity中显示；
  • Fork：Pull request相关，之后叙述。
  • Code：展示代码。
  • Issue：用于bug报告、功能添加、方向讨论等，pull request时也会创建Issue。
  • Pull request：Github重要功能，代码修改和功能讨论都可以在这里进行。
  • wiki：创建页面，通过markdown语法编写软件的使用文档等。
  • insights：透视，查看仓库的开发程度和活跃程度等。
  • settings：设置仓库，私有共有、删除仓库等。

下侧

  • commits：提交次数和提交历史均可查看。
  • branches：查看仓库的分支列表。
  • releases：对归档的文档添加标签，也可提供可用的软件。
  • contributors：对仓库进行提交的程序员的名单，pull request且被采纳。
  • create new file：网页端创建文件和添加。
  • clone or download：克隆或者复制，克隆时建议使用SSH协议，此时克隆到本地的仓库会和远程仓库的master分支内容保持一致。

文件

关于Github仓库内相关文件的操作，打开仓库中的任意一个文件即可。

  • raw：直接在浏览器显示内容，对应的URL可通过HTTPS协议下载。
  • Blame：能够按行显示提交的信息。
  • History：查看文件的历史提交信息。
  • 后面的三个按钮，电脑是表示在客户端打开的意思，edit是编辑的文件的意思，最后一个delete是删除当前文件的意思。
  • 在仓库页面按下t键，输入想要的文件名，可快速查找。
  • 在文件的URL后添加#L10~L15的标志，会高亮代码的第10到第15行，方便交流代码。

查看区别

如果差别过大则不会列出所有区别，某种意义上Github是一款优秀的源代码查看器。

  • 查看两个分支的区别：将分支名加入对应的URL里即可，/compare/4-stable...3-stable；
  • 查看与几天前的区别：/compare/master@{7.day.ago}...master，day, year, month, week都可以；
  • 查看与指定日期的区别：/compare/master@{2013-01-01}...master。

Issue

跟踪BUG和讨论的专区。用于：

  • 发现软件的BUG并报告；
  • 有事情想找软件的作者进行询问，如：如何实现XXX功能；
  • 事先列出今后准备开发的任务；

其编辑为常用的MarkDown语法，上传图片和文件直接拖入即可，在管理软件开发也有独特的应用：

  • 对仓库有读写权限的可以给每一个Issue添加标签，便于管理；
  • 添加里程碑(milestones)，距离下一个版本还有几个Issue待解决。

而在有些专门的仓库中，在提交Issue时会有专门的模板，此时在仓库的根目录下添加CONTRIBUTING.md即可，用于规范问题的提问方法。也支持Tarklist等独有语法生成复选列表：

  • 且可以选中一句话，然后按击r即可完成引用。
  • 输入:号，还可以输入表情。

通过commit操作Issue

只需要在commit按照特定格式描述信息，就可以使用bug追踪系统对Issue进行操作。如，首先手动new一个issue，然后记住他的编号，这里假设编号为12。在12这个issue里，输入的大致内容为完成数据的存储功能。那么在编写完数据存储功能后，有两种操作issue的用法：

  • git commit -m "add subfuntion: select file type when save data #12"，这样在评论中添加issue的编号后，会在issue中添加对应的描述。
  • git commit - m "fix #12"，会自动关闭12这个issue，这样就可以利用issue驱动项目的开发。

pull request

用户修改代码后向对方仓库发送请求接纳的功能，也是Github的核心功能。

  • 可以获取diff和patch格式的文件来处理pull request，此时在URL的末尾添加.diff和.patch即可。
  • 在URL末尾添加?w=1可以不显示空格的差别。
  • 将鼠标移动到代码左侧的+号，点击，可以对具体的一行代码添加注释。（查看每次的提交也可以这样）

wiki

使用简单语法编写文档，new page按钮可以创建新的标签页，也可在页面中的底部和侧部创建侧边栏。

insights

体现开发的活跃度。

  • pulse：多少人在开发，有多少issue，有多少pull requests一目了然；
  • contributors：这个仓库的开发有多少参与者；
  • community：社区文件，包括一个项目必有的7个部分；
  • traffic：有多少人浏览了这个项目，有多少人clone了这个项目；
  • commits：提交历史；
  • code frequency：该仓库的代码增加量或者删除量
  • network：以图表的形式显示包括clone仓库在内的所有分支的提交，看出每个人的工作。
  • fork：有多少人fork了你的代码。

settings

  • settings：设置仓库的默认分支。
  • features：可以移除issue等选项，则这个仓库的issue将无法启用。
  • danger zone：删除仓库等危险操作。
  • collaborators：合作者，邀请他人一起开发。如果是organization，可以创建team对整个团队赋予权限。
  • webhooks：添加钩子，让github仓库与其他服务集成。

Others

  • issue里面的消息不想继续接收可以点击unsubscribe。
  • notification里面的消息不想继续接收可以点击扩音器那个标志。
  • github pages 搭建博客。
  • github API：Web开发者难免调用的。
  • github Enterprise：无法将代码开源，但又想用github的团队可以考虑。
