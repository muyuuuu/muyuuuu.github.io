分析程序设计中的多进程

发表于 2019-02-26 更新于 2019-02-27   阅读次数：
本文字数： 6.1k 阅读时长 ≈ 6 分钟

其实在很早之前听说过程序设计中的并发编程，也就是在程序设计使使用多进程的手段。好像还在大二上学期的windows开发的课上提过这种技术，不过因为课上提供的冗长的C#程序和讲的一点道理都没有的例子，当时对这个技术毫无兴趣也没放在心上。

对这个东西突来灵感是因为某次思考，当时我在想，如果要设计一个简单的程序，这个程序分为两部分，一部分用于显示地区和温度，一个程序显示时间（精确到秒）。如果为了显示时间，这个程序会一直执行下去，也就是显示时间的程序是一个死循环。那么问题来了，程序陷入某个死循环，后面的程序该怎么执行？这个时候，多进程编程就可以派上用场了。

也就是一个进程去执行死循环，另外的进程去执行下面执行的任务，两不耽误。也是思考到这里，我也明白了一点：计算机科学系的学生也在编程，工科专业的学生会编程，如果当初我不转专业，可能一样在愉快的写代码。两者写出来的程序区别在哪里呢？可能差异在对计算机系统的理解。

通俗举例

一个通俗的例子：并发技术，就是可以让你在同一时间同时执行多条任务的技术。你的代码将不仅仅是从上到下，从左到右这样规规矩矩的一条线执行。你可以一条线在main函数里跟你的客户交流，另一条线，你早就把你外卖送到了其他客户的手里。

所以，为何需要并发？因为我们需要更强大的功能，提供更多的服务，所以并发，必不可少。

进程：

回归计算机操作系统中进程的概念：我写了一份python的程序，这个程序在系统执行的时候，操作系统给他分配了进程控制块（PCB），使用PCB来描述进程执行需要的外部数据和一些执行状况。

  • 程序是一组指令的集合，它静态存储于诸如磁盘之类的存储器里；
  • 当一个程序被操作系统执行时，它就会被载入内存空间，并在逻辑上产生一个独立的实例，这就是进程，更偏向于执行过程，是一个程序在CPU上顺序执行的活动。

学艺不精的我此时产生了一个疑问：

我写了一个程序，这个程序里面有两个子函数（$F1,F2$），我创建了两个进程（$P1,P2$），主函数执行时，$P1$去执行$F1$，$P2$去执行$F2$。假设两个进程之间没有通信和临界资源的访问冲突。

那么操作系统会为这个程序创建几个进程控制块PCB？假设此时计算机只执行这个程序，此时计算机中有几个进程？

是一个进程？因为进程是这个程序的执行过程，只有一个程序。还是两个进程？因为程序中创建了两个进程？

后来不断的google明白了：上面的$P1,P2$是从主进程生出来的子进程，所以此时的系统只有3个进程，主进程+$P1$+$P2$。

这里其实是看别人分析linux的源码才明白的。

此处感谢Liam Huang大佬的解答。Liam Huang大佬的解答，包括我的博客也在模仿Liam Huang的风格。

原谅我的弱鸡，大学听的最差的两门课就是操作系统和Liunx编程。（手动滑稽，这两门课稍微讲点正儿八经的内容我就不至于这么菜了）

线程

线程是进程内部的一个执行单元。系统创建好进程后，实际上就启动执行了该进程的主执行线程，主执行线程以函数地址形式，比如说main函数，将程序的启动点提供给Windows系统。主执行线程终止了，进程也就随之终止。所有进程有至少一个线程，但是它们可以具有多个任务。这意味着单个程序可以并发执行两个或者多个任务。

每一个进程至少有一个主执行线程，它无需由用户去主动创建，是由系统自动创建的。用户根据需要在应用程序中创建其它线程，多个线程并发地运行于同一个进程中。一个进程中的所有线程都在该进程的虚拟地址空间中，共同使用这些虚拟地址空间、全局变量和系统资源，所以线程间的通讯非常方便，多线程技术的应用也较为广泛。

但是！！！每个优点的背后绝对有缺点，这是我当初学习操作系统最大的收获，每个提出的概念背后都有一堆补丁去弥补不足，或者压根从头提出新概念，然而这个新概念也不一定有多好，只是有自己的适应场合。

多进程编程

目前是使用的python作为开发工具，Unix/Linux无疑是编写多进程的服务程序(有fork调用)正确的选择，Mac也行，Mac系统是基于BSD（Unix的一种）内核。

由于Windows没有fork调用，所以在Windows上用Python编写多进程的程序需要一个跨平台的多进程支持，multiprocessing模块就是跨平台版本的多进程模块。multiprocessing模块提供了一个Process类来代表一个进程对象。

进程通信

进程池

有关进程池的概念，就是有一个大池子，池子里面最多能执行$x$个进程，超过$x$个进程后就等待池子里面的进程执行完毕在执行。

进程池 维持执行的进程总数为processes，当一个进程执行完毕后会添加新的进程进去。 调用join之前，先调用close函数，否则会出错。执行完close后不会有新的进程加入到pool,join函数等待所有子进程结束。

apply_async() （并行执行）in parallel. Additionally, func is only executed in one of the workers of the pool.

terminate() 立刻关闭进程池

join() 主进程等待所有子进程执行完毕。必须在close或terminate()之后使用 A process can be joined many times.A process cannot join itself because this would cause a deadlock. It is an error to attempt to join a process before it has been started.

close() 等待所有进程结束后，才关闭进程池。

在这里，并不是睡眠1秒后输出一个’yes’，而是在睡眠4次之后，总共最后一起输出4个yes。如果在死循环中加入了time.sleep(1)，那么这个进程执行中不会有任何输出，也就是，子进程跑的程序会在跑完任务后一次性输出，那么这个多进程实现不了死循环里睡眠一秒咋办呢？而且虽然没有输出，假如里面有全局变量，会不会影响变量？抱歉我没测试（关于是否影响变量，建议在spyder里面测试，算了我自己测试了，变量的值不会改变）。

其实是每次 sleep 前输出 yes ，但是这个输出并不是输出到控制台，而是输出到缓冲区。缓冲区的内容如果没有被输出到控制台之前就挂掉，缓冲区里的内容就不会有机会输出了。可以用 sys.stdout.flush() 刷新缓冲区，将缓冲区的内容全部输出到控制台再 sleep 就好了。

那么如何实现在死循环的子任务中睡眠一会儿呢？多线程编程。

线程锁

由于线程会共享这个进程所有的资源，肯定会造成临界资源的冲突，比如典型的读写冲突，一个线程在读，一个线程在写，很可能造成错误。（我假设你了解点计算机，这是最基本的）

这个时候就要对这类临界资源加锁了，也就是，在所有线程写完之后在开始读，或者所有线程读完之后在开始写。看到mutex仿佛回到了当初操作系统的课程……毕竟只讲了这里。

另外就是 Python 里的 mutex 最好是结合 with 语法来用，这样省下写 m.release() 的功夫，也不会忘记。

定时器

程序一般分为两种 CPU 密集型 和 I/O 密集型。

CPU 密集型：程序比较偏重于计算，需要经常使用 CPU 来运算。例如科学计算的程序，机器学习的程序等。 I/O 密集型：顾名思义就是程序需要频繁进行输入输出操作。爬虫程序就是典型的 I/O 密集型程序。

如果程序是属于 CPU 密集型，建议使用多进程。而多线程就更适合应用于 I/O 密集型程序。
